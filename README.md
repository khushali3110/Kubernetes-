Kubernetes + EKS: Final Project Documentation
☸️ Project Overview
This project demonstrates the complete deployment lifecycle of a containerized application using Amazon EKS. It covers everything from local code development to cloud-native orchestration, highlighting key Kubernetes components like Pods, Deployments, and Services.

🧩 Core Concepts & Definitions
Kubernetes (K8s)
Kubernetes is an open-source orchestration system that automates the management of containerized apps. Think of it as a "pilot" for your containers—it ensures they are running, healthy, and scaled to handle traffic.

Amazon EKS (Elastic Kubernetes Service)
EKS is a managed service provided by AWS. It removes the complexity of setting up a Kubernetes cluster manually, allowing developers to focus on deploying applications rather than managing infrastructure.

🚀 Key Components & Functions
Pods and Deployments
A Pod is the smallest execution unit in Kubernetes where the application code resides. A Deployment acts as a manager for these Pods; it maintains the desired number of "Replicas" and handles "Self-healing"—if a Pod crashes, the Deployment automatically restarts a new one.

Services and Load Balancing
Since Pods are temporary and their IP addresses change, a Service provides a stable, permanent entry point to access the application. In this project, we use a LoadBalancer type service, which creates an AWS Load Balancer to route external internet traffic to our internal Pods.

ConfigMaps and Secrets
To keep the application secure and flexible, we use ConfigMaps for non-sensitive data (like environment names) and Secrets for sensitive information (like database passwords). This ensures that configuration is decoupled from the actual application code.

🛠️ The Deployment Flow (Step-by-Step)
Packaging: The Node.js application is converted into a portable Docker Image using a Dockerfile.

Distribution: The image is pushed to a registry like Docker Hub so the cloud cluster can download it.

Provisioning: An EKS Cluster is established on AWS to provide the necessary computing power.




⚠️ Challenges & Troubleshooting
During the implementation of this project, I encountered and resolved several complex infrastructure challenges:

1. Docker Daemon Connectivity Issues

Challenge: The terminal failed to recognize the docker command, and connectivity to the Docker Daemon was lost due to incorrect environment path variables on Windows.

Solution: Manually updated the System Environment Variables to include the Docker bin path (C:\Program Files\Docker\Docker\resources\bin) and ensured Docker Desktop was running in the background with admin privileges.

2. Secret Encoding and Decoding Mismatch

Challenge: The application was unable to read the database password from the Kubernetes Secret, causing the backend to crash.

Solution: Realized that Kubernetes requires Secrets to be Base64 encoded. I used the echo -n "password" | base64 command to properly encode the value before adding it to secrets.yaml, and used dotenv in the code to map it correctly.

3. Load Balancer Pending State

Challenge: After applying the service.yaml, the External IP stayed in the <pending> state for a long time on the EKS cluster.

Solution: Identified that the EKS cluster lacked the necessary IAM OIDC Provider and AWS Load Balancer Controller permissions. I resolved this by associating the IAM role with the service account using eksctl, which allowed AWS to provision the ELB (Elastic Load Balancer) successfully.

4. ConfigMap Synchronization Delay

Challenge: When updating the APP_MODE in configmap.yaml, the changes were not reflecting in the running Pods immediately.

Solution: Learned that Kubernetes does not automatically restart Pods when a ConfigMap is updated. I implemented a Rolling Restart using the command kubectl rollout restart deployment/intern-app-deployment to force the Pods to pick up the new configuration.

Orchestration: The YAML Manifests (Deployment, Service, Secret) are applied using the kubectl command.

Validation: The final application is accessed via the External IP generated by the AWS Load Balancer, confirming a successful full-project flow.
